<objective>
Fix the critical shutdown URL saving bug where SIGTERM/SIGINT during scraping saves zero URLs.

Purpose: The scraper finds URLs during its scroll loop, but the shutdown handler only has access to an empty Set because state.urls is populated after runScraper() returns. Non-daemon shutdown handlers are also registered after scraping completes, so signals during scraping are uncaught.

Output: Both daemon and non-daemon modes save all collected URLs when interrupted during scraping.
</objective>

**Step 1: Add targetUrls to ScraperOptions (src/types.ts)**

Add optional `targetUrls?: Set<string>` field to the `ScraperOptions` interface. This allows the caller to pass a shared mutable container that the scraper populates during extraction.

**Step 2: Wire scraper to populate targetUrls (src/scraper.ts)**

In `runScraper`, after `setupGraphQLInterceptor(page, profileUrls, logger)` at line 73, if `options.targetUrls` is provided, add a mechanism to also add extracted URLs to `options.targetUrls`. Two approaches (choose one):

Option A (preferred — single source of truth): Use `options.targetUrls` as the primary Set and pass it to `setupGraphQLInterceptor` instead of the local `profileUrls`. The local `profileUrls` variable becomes a reference to `options.targetUrls` when provided. This way the interceptor writes directly to the shared container.

Implementation: After line 49 (`const profileUrls = new Set<string>()`), add:
```
const targetSet = options.targetUrls ?? profileUrls;
```
Then pass `targetSet` to `setupGraphQLInterceptor` instead of `profileUrls`. Replace all references to `profileUrls` with `targetSet` in the rest of the function. The function still returns `targetSet` (which is the same object as `options.targetUrls` when provided, or the local Set when not).

This means `state.urls` in index.ts IS the Set the scraper writes to — no copying needed.

**Step 3: Register non-daemon shutdown BEFORE runScraper (src/index.ts)**

Move the non-daemon shutdown handler registration (lines 157-182) to BEFORE the `runScraper()` call (before line 154). Both daemon and non-daemon modes should register handlers before scraping starts.

The non-daemon shutdown handler should use `state.urls` (the shared container) instead of the local `urls` variable. Since `state.urls` is now populated during scraping via `targetUrls`, the closure will have access to all URLs found so far.

Concretely:
1. Remove the non-daemon shutdown block from lines 157-182
2. After the daemon shutdown block (line 138), add the non-daemon shutdown handler with the same pattern but using `state.urls`:
   - Register `SIGINT` and `SIGTERM` handlers that call `saveUrlsToFile(outputFile, state.urls)`
   - Include browser cleanup
   - Use a `shuttingDown` flag to prevent double execution
3. Change the `runScraper` call to pass `targetUrls: state.urls` in the options
4. Remove the `state.urls = urls;` assignment at line 155 (no longer needed since targetUrls IS state.urls)
5. Keep the `urls` return value from `runScraper` for the completion log and final save

The resulting flow:
```
state.urls = new Set()
register shutdown handlers (both daemon and non-daemon) using state.urls
urls = await runScraper({ ... targetUrls: state.urls })
// urls === state.urls (same object reference)
saveUrlsToFile(outputFile, urls)  // final save
webhook(...)
```

**Step 4: Verify backward compatibility**

Ensure that when `targetUrls` is NOT provided, the scraper creates its own internal Set and returns it (existing behavior unchanged). This matters for any callers that don't use the shared container pattern.

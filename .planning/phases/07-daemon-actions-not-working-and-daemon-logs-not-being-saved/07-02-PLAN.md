**Part A: Replace setTimeout with polling loop in handleStop (UAT gap 1)**

The current `handleStop()` sends SIGTERM then uses a fixed 2-second `setTimeout` to check if PID file was cleaned up. This is fire-and-forget — if the daemon takes longer than 2s to exit, the PID file is never removed.

Replace the setTimeout block (lines 32-38 in daemon-actions.ts) with a polling loop:
- After sending SIGTERM, poll `isProcessRunning(pid)` every 200ms for up to 5 seconds (25 iterations)
- If process exits within timeout: call `removePidFile()` and print "Daemon stopped"
- If process still running after 5s: print "Warning: daemon process still running after stop signal" and print the PID
- Also remove the PID file immediately after SIGTERM is sent if the process is already gone (race condition where process exits before we poll)
- Import `removePidFile` (already imported) and use it for cleanup
- The polling approach ensures the PID file is cleaned up regardless of daemon exit timing

The handleStop function should become async and return a Promise so the main() caller can await it. Update the main() switch to use `await handleStop()`.

**Part B: Move setupDaemonShutdown before runScraper (UAT gap 1 — race window)**

In src/index.ts, `setupDaemonShutdown` is registered at line 136-149, AFTER `runScraper()` returns at line 133. This creates a race window where SIGTERM arrives before the handler is registered.

Move the `if (process.env.SCRAPER_DAEMON_CHILD === '1')` block (lines 136-149) to BEFORE the `runScraper(options)` call (line 133). The shutdown handler references `urls` and `browserRef`, so:
- `urls` is used in `saveState` callback — this is a closure, so it captures the reference. Since `urls` is declared at line 133 as `const urls = await runScraper(options)`, it's not available before runScraper. Instead, use a mutable container: `const state = { urls: new Set<string>() }` declared before runScraper, then update `state.urls = urls` after runScraper returns. The shutdown handler's `saveState` closure reads `state.urls`.
- `browserRef` is used in `cleanup` callback — it's already declared at line 117 as `let browserRef = null` and updated by the `onBrowserReady` callback. Since the callback fires during runScraper, the reference is valid by the time SIGTERM arrives. No change needed for browserRef.
- Move the setupDaemonShutdown call to just before runScraper, using the `state.urls` pattern.

The flow becomes:
1. Declare `const state = { urls: new Set<string>() }`
2. Register setupDaemonShutdown (refs state.urls and browserRef)
3. Call `const urls = await runScraper(options)`
4. After runScraper: `state.urls = urls` (so shutdown handler has the final data)
5. Continue with rest of function (log, save, webhook)

Do NOT move the non-daemon shutdown handler (lines 150-174) — that one correctly runs after runScraper because it's not daemon-mode.

**Part A: Fix child-process log writing (UAT gap 2)**

The current implementation creates `logStream` in the parent process and pipes `child.stdout/stderr` to it. But `index.ts` calls `process.exit(0)` at line 68 after `startDaemon()` returns, killing the parent and destroying the pipe. The child continues detached but its pino logger writes to `process.stdout` (broken pipe).

The fix: configure the child process to write its own logs directly to the `.daemon.log` file instead of relying on parent-to-child piping.

In src/daemon.ts `startDaemon()`:
- Instead of creating logStream in the parent (lines 86-88), pass the log file path as an environment variable to the child process
- Add to the fork env: `SCRAPER_LOG_FILE: resolvedLogPath` (where resolvedLogPath is the resolved LOG_FILE path from Part A of Task 2)
- Remove the `logStream`, `child.stdout?.pipe(logStream)`, `child.stderr?.pipe(logStream)` lines (86-88) — they're useless since parent exits
- Keep `stdio: ['ignore', 'pipe', 'pipe', 'ipc']` — but now pipe to parent is unused (could change to `'ignore'` but keeping pipe doesn't hurt)

In src/index.ts (child process side, after the daemon check):
- At the top of the child process path (after the `if (argv.daemon && process.env.SCRAPER_DAEMON_CHILD !== '1')` block exits), check for `SCRAPER_LOG_FILE` env var
- If present, create a WriteStream and redirect the pino logger output to it
- Update `createLogger()` call (line 34) to accept an optional file destination
- Alternative simpler approach: In `src/logger.ts`, if `SCRAPER_LOG_FILE` env var is set, configure pino to write to that file instead of stdout. This is cleaner because it keeps the change in one place.

Best approach — modify src/logger.ts:
- Read `process.env.SCRAPER_LOG_FILE` 
- If set, create a pino destination that writes to that file: `pino.destination({ dest: process.env.SCRAPER_LOG_FILE, sync: false })`
- Pass this destination to pino: `pino({ level: 'info' }, destination)`
- If not set, use current behavior (stdout)

This is the cleanest fix — one change in logger.ts, the child process automatically writes logs to the file specified by the env var, and the parent can exit safely.

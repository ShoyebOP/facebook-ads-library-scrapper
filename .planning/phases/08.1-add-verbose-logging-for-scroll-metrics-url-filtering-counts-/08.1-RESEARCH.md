# Phase 08.1: Add verbose logging for scroll metrics, URL filtering counts, and stopping reason diagnostics - Research

**Researched:** 2026-07-08
**Domain:** Structured logging, metrics tracking, diagnostic output
**Confidence:** HIGH

## Summary

This phase adds detailed diagnostic logging to help debug scraper behavior by tracking scroll metrics (scroll height, URLs found per scroll, filtered count, running unique total), URL filtering breakdown per scroll, and a full diagnostic snapshot when the scraper stops. The implementation leverages the existing pino structured logging infrastructure with child loggers and extends the scroll loop in `src/scraper.ts` and the GraphQL interceptor in `src/extractor.ts`.

**Primary recommendation:** Add per-scroll metrics logging at INFO level with structured fields, track filtered URL counts by comparing Set size before/after extraction, and create a stopping reason diagnostic snapshot with reason string, final stats, total time, and scroll count.

### Pattern 1: Per-Scroll Metrics Logging
**What:** Log scroll height, URLs found, filtered count, and running total after each scroll
**When to use:** In the scroll loop after each scroll operation completes
**Example:**
```typescript
// Source: Context7/official docs URL
// After scroll and DOM cleanup
const scrollHeight = await page.evaluate(() => window.scrollY);
const urlsFound = targetSet.size - before;
const filteredCount = urlsFound > 0 ? 0 : 0; // Will be tracked from extractor
const runningTotal = targetSet.size;

scrollLogger.info({
    scrollHeight,
    urlsFound,
    filteredCount,
    runningTotal,
}, `Found: ${urlsFound}, Filtered: ${filteredCount} (dups), Unique: ${runningTotal}, Total: ${runningTotal}`);
```

### Pattern 2: Filtered URL Count Tracking
**What:** Track how many URLs were filtered (duplicates) during extraction
**When to use:** In the GraphQL interceptor when extracting URLs from responses
**Example:**
```typescript
// Source: Context7/official docs URL
// In extractor.ts setupGraphQLInterceptor
page.on('response', async (response: Response) => {
    try {
        if (response.status() === 200 && response.url().includes('graphql')) {
            const json = await withTimeout(response.json(), 15000);
            const beforeSize = profileUrls.size;
            extractProfileUrls(json, profileUrls);
            const filteredCount = (profileUrls.size - beforeSize) > 0 ? 0 : 0;
            // Track filtered count for metrics
        }
    } catch (e) {
        // Error handling
    }
});
```

### Pattern 3: Stopping Reason Diagnostic
**What:** Log full diagnostic snapshot when scraper stops with reason, final stats, total time, scroll count
**When to use:** In the pipeline orchestrator after runScraper completes
**Example:**
```typescript
// Source: Context7/official docs URL
// In index.ts after runScraper
const result = await runScraper(options);
const elapsedMs = Date.now() - startTime;
const elapsedMin = Math.floor(elapsedMs / 60000);
const elapsedSec = Math.floor((elapsedMs % 60000) / 1000);

logger.info({
    reason: result.reason,
    totalTime: `${elapsedMin}m ${elapsedSec}s`,
    scrolls: result.scrollCount,
    uniqueUrls: result.urls.size,
}, `Stopped: ${result.reason} (${result.urls.size}/${result.maxUrls}). Total time: ${elapsedMin}m ${elapsedSec}s. Scrolls: ${result.scrollCount}. Unique URLs: ${result.urls.size}.`);
```

### Anti-Patterns to Avoid
- **String concatenation in log messages:** Use structured fields instead of string interpolation for queryability
- **Logging inside tight loops:** Per-scroll logging is acceptable, but avoid logging per-URL extraction
- **Missing error context:** Always include error objects with `err` key for pino serialization
- **Hardcoded log levels:** Use child loggers with module context instead of hardcoded level checks

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Structured logging | Custom console.log formatting | Pino with child loggers | High performance, async, built-in redaction |
| Log formatting | Manual string building | pino-pretty | Consistent formatting, color support, daemon mode |
| URL deduplication tracking | Custom Set wrapper with counters | Track before/after Set size | Simpler, less error-prone, follows existing patterns |
| Time formatting | Manual milliseconds calculation | Existing elapsed time calculation | Already implemented in index.ts |

**Key insight:** The existing pino infrastructure with child loggers and structured fields provides everything needed for verbose logging. Custom solutions would duplicate functionality and introduce maintenance overhead.

## Common Pitfalls

### Pitfall 1: Logging Performance Impact
**What goes wrong:** Adding too much logging in the scroll loop slows down scraping
**Why it happens:** Pino is fast but not free; logging per-URL extraction would be excessive
**How to avoid:** Log per-scroll metrics (not per-URL), use structured fields instead of string concatenation, avoid logging in tight loops
**Warning signs:** Scraper becomes noticeably slower, scroll interval increases

### Pitfall 2: Missing Error Context
**What goes wrong:** Error logs lose stack traces and context
**Why it happens:** Logging error messages as strings instead of using pino's error serializer
**How to avoid:** Always pass errors with `err` key: `logger.error({ err }, 'message')`
**Warning signs:** Debugging becomes difficult, no stack traces in logs

### Pitfall 3: Inconsistent Log Format
**What goes wrong:** Log messages become hard to parse and query
**Why it happens:** Mixing string concatenation with structured fields
**How to avoid:** Use structured fields for all metrics, keep messages concise
**Warning signs:** Log aggregation tools can't parse fields, manual grep required

### Pitfall 4: Missing Scroll Height on Error
**What goes wrong:** Scroll height tracking fails when page doesn't scroll
**Why it happens:** `window.scrollY` may return 0 or throw on error pages
**How to avoid:** Wrap scroll height evaluation in try/catch, default to 0 on error
**Warning signs:** NaN or undefined values in logs, missing metrics
